# 监督微调损失

## 1. 因果语言模型在学习什么

一段 token 序列记为：

```text
x_1, x_2, ..., x_T
```

因果语言模型只能看当前位置之前的 token，因此整段文本的概率分解为：

```math
p_\theta(x_1, ..., x_T)
= \prod_{t=1}^{T} p_\theta(x_t \mid x_{<t})
```

模型在位置 `t - 1` 输出词表上的 logits，用 softmax 得到下一个 token `x_t` 的概率。训练的基本目标是最大化正确 token 的概率，等价于最小化负对数似然：

```math
\mathcal{L}_{\mathrm{NLL}}
= -\frac{1}{T}\sum_{t=1}^{T}
\log p_\theta(x_t \mid x_{<t})
```

这个项目做的是监督微调，不改变“预测下一个 token”这件事，只改变哪些 token 参与损失。

## 2. 为什么只训练 assistant 回复

一条对话被渲染成：

```text
system: 你是严谨助手
user: 什么是 LoRA？
assistant: LoRA 是一种参数高效微调方法。
```

system 和 user 是条件，assistant 是应学习的答案。定义掩码 `m_t`：

```math
m_t =
\begin{cases}
1, & x_t \text{ 属于 assistant 片段} \\
0, & x_t \text{ 属于 system 或 user 片段}
\end{cases}
```

项目实际优化的掩码交叉熵为：

```math
\mathcal{L}_{\mathrm{SFT}}
= -\frac{1}{\sum_t m_t}
\sum_{t=1}^{T} m_t
\log p_\theta(x_t \mid x_{<t})
```

因此模型学的是“给定系统提示和用户问题，接下来如何写 assistant 回复”，而不是复述用户问题。

## 3. `-100` 在代码中的含义

Transformers 的因果语言模型通常接收 `input_ids` 和同长度的 `labels`。模型内部会自动把 labels 左移一位来和下一个 token 的预测对齐。

在 PyTorch 的交叉熵中，标签为 `-100` 的位置使用默认 `ignore_index`，不计入损失。项目构造的逻辑是：

```text
assistant token: label = 该 token id
system/user token: label = -100
padding token: label = -100
```

`trainer/dataset.py` 的 `build_token_spans()` 会对每条消息逐步应用 chat template，再用相邻两次结果的前缀差定位这条消息新增的 token。assistant 新增片段保留标签，其余片段全部填 `-100`。

`trainer/collator.py` 再按 batch 的最长长度做动态 padding，并给 labels 补 `-100`。这样 padding 不会被模型当成需要预测的答案。

## 4. chat template 为什么是损失的一部分

模型并不直接理解 Python 字典中的 `role` 和 `content`，它看到的是 tokenizer 渲染后的 token 序列。若训练时使用一种角色标记、推理时又换成普通字符串拼接，条件分布已经变了：

```math
p_\theta(\text{answer} \mid \text{训练模板})
\ne
p_\theta(\text{answer} \mid \text{另一种模板})
```

所以本项目训练和推理都调用 `tokenizer.apply_chat_template()`。模型自带模板优先；缺失时 `model/tokenizer.py` 提供一个兼容 Qwen 消息格式的后备模板。

## 5. 截断为何会破坏训练

设原始长度为 `T`，配置长度为 `L`。当 `T > L` 时，样本会被截断。若保留的 `L` 个 token 全是 system/user 上下文，则：

```math
\sum_t m_t = 0
```

损失没有任何有效项，训练这条样本没有意义。项目先按 tokenizer 的截断方向截断；若 assistant 全部被裁掉，会回退保留尾部 token。若仍然没有 assistant 标签，直接报错，提示清洗数据或提高 `max_length`。
