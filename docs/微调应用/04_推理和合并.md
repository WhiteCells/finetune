# 推理和合并

## 1. 直接加载 adapter 推理

打开 `inference.py`，修改 `RUN_CONFIG`：

```python
RUN_CONFIG = InferenceConfig(
    model_name_or_path="./models/Qwen3-4B-Instruct-2507",
    adapter_path="outputs/qwen3-4b-lora-exp001",
    prompt="请用一句话解释 LoRA。",
    max_new_tokens=128,
    do_sample=False,
)
```

运行：

```bash
uv run inference.py
```

`do_sample=False` 适合固定题集比较，因为每次输出更稳定。需要更有变化的回答时改为：

```python
do_sample=True,
temperature=0.7,
top_p=0.9,
```

推理脚本自动使用 tokenizer 的 chat template，并将输入放到模型首个参数所在设备；基座模型采用 `device_map="auto"` 加载。

## 2. 保存回答

在同一个配置中加入：

```python
output_file="outputs/sample_answer.txt",
```

脚本仍会打印回答，同时写出该文件。

## 3. 合并 LoRA

adapter 小、可继续训练，但推理时依赖 PEFT 和基座模型。部署方需要普通 Transformers 模型目录时，打开 `merge_lora.py`：

```python
RUN_CONFIG = MergeConfig(
    model_name_or_path="./models/Qwen3-4B-Instruct-2507",
    adapter_path="outputs/qwen3-4b-lora-exp001",
    output_dir="outputs/qwen3-4b-merged",
)
```

运行：

```bash
uv run merge_lora.py
```

输出目录会包含完整模型权重、tokenizer 和 `merge_summary.json`。合并会占用接近完整模型的内存和磁盘空间，因此应在有足够资源的机器上执行。

## 4. 选择哪种产物

| 需求 | 应保留的产物 |
| --- | --- |
| 多个任务共享一个基座模型 | adapter |
| 继续训练 | adapter 和 checkpoint |
| 单独交付或不使用 PEFT 部署 | 合并模型 |
| 可回退、可复现实验 | adapter、checkpoint、`config_snapshots/` |

无论是否合并，都应保留原始 adapter。合并模型不是继续 LoRA 训练的首选输入。

采样参数背后的公式见 [生成和采样](../公式细节/04_生成和采样.md)。
